## Context

Slice V2 of the public developer platform (#194). V1 (#318, merged) created the `api_keys` table and CRUD UI. V2 teaches `AuthGuard` to validate Bearer tokens against that table and construct a synthetic `AuthenticatedSession` for API consumers.

Promoted from: `artifacts/frames/319-auth-guard-api-key-fallback-frame.mdx`
Parent spec: N4, S1 from `artifacts/specs/194-public-api-developer-platform.mdx`

## Goal

Extend `AuthGuard.canActivate()` with an API key Bearer token fallback so that external clients can authenticate programmatically using `Authorization: Bearer sk_live_xxx`, with the same `@Permissions()` semantics as session users.

## Users

- **External developers / API consumers** — send `Authorization: Bearer <api_key>` and expect `@Permissions()` to work identically.
- **Internal CI/scripts** — use API keys for machine-to-machine calls.
- **Operators** — superadmins must NOT bypass scope checks via API key auth (existing superadmin session bypass is explicitly suppressed).

## Expected Behavior

### Happy path — API key auth

1. Client sends `Authorization: Bearer sk_live_<random>` to any protected endpoint.
2. Guard extracts the token; recognises `sk_live_` prefix → enters API key path.
3. Guard calls `ApiKeyService.validateBearerToken(token)` which:
   a. Reads `keyHash` and `keySalt` from all non-revoked rows (scoped query by partial prefix for efficiency if feasible, otherwise full scan with early exit).
   b. Computes `hmacHash(token, row.keySalt)` for each candidate; compares to `row.keyHash`.
   c. Returns the matching row or throws `API_KEY_INVALID`.
4. Guard verifies:
   - Not revoked (`revokedAt IS NULL`) → else throws `API_KEY_REVOKED` (401).
   - Not expired (`expiresAt IS NULL OR expiresAt > now`) → else throws `API_KEY_EXPIRED` (401).
5. Guard fetches org's current permissions via `PermissionService.getPermissions(row.userId, row.tenantId)`.
6. Guard computes `effectiveScopes = row.scopes ∩ orgPermissions` (runtime intersection prevents stale elevated access).
7. Guard constructs synthetic `AuthenticatedSession`:
   ```ts
   {
     user: { id: row.userId, role: <fetched from DB> },
     session: { id: row.id, activeOrganizationId: row.tenantId },
     permissions: effectiveScopes,
     actorType: 'api_key',
     apiKeyId: row.id,
   }
   ```
8. Guard fires `ApiKeyService.touchLastUsedAt(row.id)` — **fire-and-forget**, no `await`.
9. Existing `checkPermissions()` runs against `effectiveScopes`; `checkRoles()` is **bypassed** for API key auth.
10. Superadmin enforcement: `checkPermissions()` skips its early-return for superadmins when `session.actorType === 'api_key'` → scope check always applies.
11. Request proceeds.

### `@RequireApiKey()` at controller level

Routes decorated with `@RequireApiKey()`:
- `@AllowAnonymous()` has NO effect — the guard still requires authentication.
- Session auth (actorType `'user'`) is rejected with `401 API_KEY_REQUIRED`.
- Only API key auth (`actorType === 'api_key'`) proceeds.

### Fallback ordering

```
canActivate():
  1. isRequireApiKey?        → if yes, skip @PUBLIC check
  2. isPublic?               → return true (only if !requireApiKey)
  3. hasBearerToken?         → Authorization: Bearer header present?
     a. hasApiKeyPrefix?     → token starts with 'sk_live_'
        → ApiKeyService.validateBearerToken() → synthetic session OR throw API_KEY_*
     b. else (unknown Bearer format)
        → fall through to session path
  4. else (no Bearer token)  → try session path (existing behavior)
  5. isOptional?             → if !session, return true
  6. !session                → throw UnauthorizedException
  7. checkSoftDeleted        → session auth only (skip for api_key actorType)
  8. checkRoles              → skip for api_key actorType
  9. checkOrgRequired        → run for all
  10. checkPermissions       → run for all; superadmin bypass suppressed for api_key
```

Non-`sk_live_` Bearer tokens (e.g. JWTs) fall through to the session auth path, which rejects them as invalid — correct outcome, no extra handling needed.

### Error responses

| Condition | HTTP | ErrorCode |
|-----------|------|-----------|
| Token not found / hash mismatch / no `lastFour` match | 401 | `API_KEY_INVALID` |
| Token found but revoked | 401 | `API_KEY_REVOKED` |
| Token found but expired | 401 | `API_KEY_EXPIRED` |
| Scope check fails | 403 | `API_KEY_SCOPE_DENIED` |
| `@RequireApiKey()` + session auth | 401 | `API_KEY_REQUIRED` |
| Key owner user deleted (shouldn't happen — cascade revokes on delete) | 401 | `API_KEY_INVALID` |

### Key design decisions

1. **Hash lookup strategy** — `keyHash = HMAC(token, salt)`, so we can't query by hash without knowing the salt. Approach: extract the last 4 chars of the bearer token and query `WHERE lastFour = ? AND revokedAt IS NULL AND (expiresAt IS NULL OR expiresAt > now)`. This narrows to a small candidate set. Then HMAC-verify `hmacHash(token, candidate.keySalt)` against each `candidate.keyHash`. First match wins; no match → `API_KEY_INVALID`. Hash collisions on `lastFour` are rare but possible (4 chars ≈ 1.7M combinations) — iteration handles them correctly. If somehow two rows produce the same HMAC (cryptographic collision, < 2^-256 probability), log the anomaly and use the first match.

2. **No auth dependencies in `ApiKeyService`** — `validateBearerToken()` is a pure lookup/hashing method. All auth-aware logic (permission intersection, synthetic session construction, superadmin enforcement) lives in `AuthGuard`. This prevents circular dependencies and keeps the service reusable (rate limiter, CLI auth, etc.). If a circular dep is ever unavoidable, use `forwardRef()` — but the current design avoids it.

3. **User role fetch** — `AuthenticatedSession.user.role` is required. The `apiKeys` table doesn't store it. Use a JOIN on `users` in the validation query for atomicity. If the user row is missing (deleted + cascade not yet fired), return `API_KEY_INVALID`.

4. **`actorType` in `AuthenticatedSession`** — the field already exists in `types.ts` as optional. No type change needed; guard just sets it.

5. **Slice 2 depends on Slice 1** — `@RequireApiKey()` enforcement requires the synthetic session path (Slice 1) to be functional. Implement in order.

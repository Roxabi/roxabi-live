## Context

Sub-issue of #400 (code quality gaps from FE/BE standards audit). One of three original items
(dual permission systems) was already resolved before this spec. Two remain.

Source: [Frame #409](../frames/409-frontend-architecture-gaps.mdx)

## Goal

Extract raw `fetch` calls from `profile.lazy.tsx` into a `lib/profile.ts` module (consistent
with the `lib/apiKeys.ts` pattern), and promote `ApiKey*` types from `apps/web` to `packages/types`
so frontend and backend share a single type contract.

## Users

- **Frontend developers** — profile page becomes testable in isolation; raw fetch calls consolidated.
- **Backend developers** — shared ApiKey type contract eliminates silent drift risk.

## Expected Behavior

### Item A: profile.ts lib extraction

`apps/web/src/lib/profile.ts` is created with two exported functions:

```ts
export async function getProfile(fallbackName: string, fallbackId: string): Promise<ProfileData>
export async function updateProfile(payload: UpdateProfilePayload): Promise<void>
```

Both functions return raw `UserProfile` / void — the existing `parseProfileResponse()` helper
that transforms the API response into UI form state remains in `profile.lazy.tsx` (it is UI
transformation logic, not a reusable HTTP abstraction).

`fetchProfile` (module-level raw GET) and the inline `PATCH fetch` inside `handleSave` are removed
from `profile.lazy.tsx`. The component imports and calls `lib/profile.ts` functions instead.

`UserProfile` and `UpdateProfilePayload` are already in `packages/types/src/profile.ts` — the new
`lib/profile.ts` imports from there. No type duplication. Error propagation follows existing
pattern: `getProfile` throws on `!res.ok`; `handleSave` continues to catch in the component.

### Item B: ApiKey types in packages/types

A new file `packages/types/src/api-keys.ts` is created with the 5 types currently local to
`apps/web/src/lib/apiKeys.ts`:

```ts
export type ApiKey = { ... }
export type CreateApiKeyRequest = { ... }
export type CreateApiKeyResponse = { ... }
export type ListApiKeysResponse = { ... }
export type RevokeApiKeyResponse = { ... }
```

`packages/types/src/index.ts` re-exports `from './api-keys'` inserted alphabetically (after `./api`,
before `./audit`).

`apps/web/src/lib/apiKeys.ts` drops the local type definitions and imports from `@repo/types`.

Backend type shapes verified — serialized JSON responses match frontend string types for date fields
(backend returns `Date` objects, JSON serialization produces ISO strings → frontend `string | null`
is correct). No backend changes required.

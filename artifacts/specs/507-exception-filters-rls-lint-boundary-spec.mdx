## Context

Part of #501 (Slice 6). Slices 4–5 established the repository pattern with Drizzle ORM adapters. This slice enforces the architectural boundaries introduced by that work and cleans up remaining exception handling debt.

**Key constraint:** No ESLint. Project uses Biome. Custom lint rules are Bun TS scripts (grep-based pattern matching), integrated into CI and pre-push hooks.

## Goal

Enforce data-access boundaries via automated lint scripts, standardize RLS-bypass annotations, and extract the last domain-specific exception from the global filter — so architectural conventions are machine-verified, not just documented.

## Users

- **Primary:** Backend developers — get lint feedback when bypassing the repository pattern; cleaner exception handling per module.
- **Secondary:** Frontend developers — preventive lint rule stops accidental `@repo/types/api` imports in web.

## Expected Behavior

### Exception Filters

The `AllExceptionsFilter` currently has two `instanceof` checks: `ThrottlerException` and `HttpException`. The codebase already has extensive per-domain `@Catch` filters for every module (user, org, RBAC, tenant, consent, API key, admin). The only remaining extraction is `ThrottlerException` → a dedicated `ThrottlerExceptionFilter`. No other domain exceptions remain in the global filter — this was confirmed via codebase audit.

The new `ThrottlerExceptionFilter` must replicate the **full behavior** of the existing `handleThrottlerException()` private method:
- `sendErrorResponse()` with `errorCode: RATE_LIMIT_EXCEEDED`
- `Retry-After` header
- Conditional `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers (suppressed for `AUTH_SENSITIVE_PATHS`)
- Structured warn log (`RATE_LIMIT tracker= path= tier=`)
- `ClsService` injection for correlation ID

The filter imports `AUTH_SENSITIVE_PATHS` and `ThrottlerMeta` from the throttler module. After extraction, `AllExceptionsFilter` drops the `ThrottlerException` import and `handleThrottlerException()` method — only `HttpException` check (legitimate fallback) and generic catch-all remain.

**Registration:** Register in `ThrottlerConfigModule` (follows the existing per-domain pattern where each module registers its own filters), not `app.module.ts`.

### RLS-BYPASS Annotations

Every non-repository service that injects `DRIZZLE` directly gets a standardized inline comment on the `@Inject(DRIZZLE)` line:

```typescript
@Inject(DRIZZLE) private readonly db: DrizzleDB, // RLS-BYPASS: <reason>
```

**15 files** need annotations (2 already have them):

| Group | Files | Count |
|-------|-------|-------|
| admin/** | adminMembers.service, adminUsers.service, adminUsers.query, adminUsers.lifecycle, adminOrgs.service, adminOrgs.query, adminOrgs.deletion, adminInvitations.service, adminAuditLogs.service | 9 |
| Non-admin | permission.service, auth.service ✓, tenant.service, tenant.interceptor, purge.service ✓, gdpr.service | 6 |

✓ = already has `// RLS-BYPASS` comment. 13 files need new annotations.

Repository files (`*.repository.ts`) are exempt — they are the correct pattern, not bypasses.

### Lint Scripts

Two Bun TS scripts run in CI and pre-push hooks:

1. **`scripts/lint/check-drizzle-injection.ts`** — Scans `apps/api/src` for `@Inject(DRIZZLE)` outside allowed paths. Exits non-zero on violation. Paths resolved relative to `apps/api/src` for precision.

   Allowed path patterns:
   - `**/*.repository.ts` — repository pattern (correct usage)
   - `admin/**` — admin services (RLS-bypass by design)
   - `rbac/permission.service.ts` — RBAC resolution requires cross-tenant queries
   - `auth/auth.service.ts` — better-auth adapter, pre-tenant context
   - `tenant/tenant.service.ts` — manages RLS via `SET LOCAL ROLE`
   - `tenant/tenant.interceptor.ts` — org lookup during tenant resolution
   - `purge/purge.service.ts` — cron job, no tenant context
   - `gdpr/gdpr.service.ts` — data export across tenants

   **Spec decision:** The issue listed only `*.repository.ts`, `admin/**`, `permission.service.ts`, `auth.service.ts` as allowed. The 4 additional paths (`tenant.service`, `tenant.interceptor`, `purge.service`, `gdpr.service`) were added based on codebase audit — these are legitimate DRIZZLE injectors with documented RLS-bypass reasons.

2. **`scripts/lint/check-type-imports.ts`** — Scans for `@repo/types/api` imports in `apps/web` and `@repo/types/ui` imports in `apps/api`. Exits non-zero on violation.

Both scripts output clear error messages with file path and line number.

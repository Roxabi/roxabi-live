## Problem

API keys currently share the same IP-based rate limiting as browser sessions. This means a single API key used from multiple servers gets generous limits (one per IP), while a heavy-traffic integration from one IP can starve other users on the same address. There is no per-key tracking, no rate limit headers for clients to self-regulate, and no distinct 429 error code for API key exhaustion.

V2 (#319, merged) established `req.apiKey` in the auth guard. This slice adds a dedicated `api` throttler tier that tracks by key ID instead of IP, returns standard `X-RateLimit-*` headers, and emits a typed 429 when exceeded.

## Who

- **Primary:** API consumers (third-party developers integrating via API keys) — need predictable, per-key limits and headers to implement backoff
- **Secondary:** Platform operators — need per-key isolation so one noisy consumer doesn't degrade others

## Constraints

- Must reuse existing `CustomThrottlerGuard` + `ThrottlerConfigModule` + Upstash Redis storage (from #53)
- Default: 100 req / 60s (configurable via `RATE_LIMIT_API_TTL`, `RATE_LIMIT_API_LIMIT` env vars)
- Tracking key: `apikey:${keyId}` — per-key, not per-IP
- Non-API-key requests must be completely unaffected by the `api` tier
- Response headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- 429 response: `API_KEY_RATE_LIMITED` error code + `Retry-After` header

## Out of Scope

- Per-plan tiered limits (different limits per subscription tier) — future slice
- Rate limit dashboard / analytics UI
- Burst / sliding window algorithms — uses existing fixed-window from `@nestjs/throttler`

## Complexity

**Tier: F-lite** — Single backend domain, clear acceptance criteria, extends existing throttler infrastructure with a new tier entry and guard logic branch.

- Files: `customThrottler.guard.ts`, `throttler.module.ts`, plus tests (~3-5 files)
- No new architectural patterns — extends existing `throttlers[]` array
- Dependency #319 already merged

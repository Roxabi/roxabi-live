## Problem

External developers and AI agents need versioned, key-authenticated REST endpoints to interact with the Roxabi platform programmatically. The auth guard (#319) now authenticates API keys, but there are no public-facing controllers — callers have no endpoints to hit.

Without dedicated V1 controllers, consumers would need to call internal admin endpoints directly, coupling them to internal response shapes and risking breaking changes when internal services evolve.

## Who

- **Primary:** External developers integrating with the Roxabi API via `Authorization: Bearer sk_live_xxx`
- **Secondary:** AI agents (future V6), CLI (future V5) — both consume the same `/api/v1/` endpoints

## Constraints

- `@RequireApiKey()` at controller level on all V1 controllers (enforced by #319 auth guard)
- Purpose-built response DTOs — decoupled from internal service shapes (per parent spec technical decision)
- Error envelope: `{ error: { code, message, statusCode } }`
- Delegates to existing services: `OrganizationService`, `AdminMembersService`, `AdminInvitationsService`, `RbacService`, `UserService`
- Dual Swagger: existing internal `/api/docs` unchanged + new public `/api/v1/docs` with `addApiKey()` security scheme
- Depends on #319 (merged, PR #448)

## Out of Scope

- Rate limiting per API key (#321, V4)
- CLI codegen (#322, V5)
- AI tool registry (#325, V6)
- HATEOAS links
- Pagination (internal services already paginate — V1 passes through)
- API versioning strategy beyond `/v1/` prefix

## Complexity

**Tier: F-lite** — Single domain (API), clear scope from parent spec, known NestJS patterns (controllers + DTOs + Swagger). No new architecture, no unknowns.

- Files: ~15 (5 controllers + 5 DTO modules + Swagger config + module registration + tests)
- Risk: Low — thin controllers delegating to existing services
- Architecture: New `v1/` module tree, but follows existing NestJS conventions
- Domains: 1 (API only)

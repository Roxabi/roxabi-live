## Problem

The API's `AllExceptionsFilter` has grown into a monolithic `instanceof` chain that handles every domain-specific error type. This makes the filter fragile, hard to test, and couples it to every module in the application. NestJS's `@Catch()` decorator system is designed for per-domain filters, but we aren't using it.

Additionally, the repository pattern (slices 4–5) introduced a clear data-access boundary, but nothing enforces it — developers can still `@Inject(DRIZZLE)` directly in services outside the allowed paths. Similarly, `@repo/types` has been split into `shared/api/ui` sub-paths, but no tooling prevents cross-boundary imports (web importing API types, API importing UI types).

Without enforcement, these architectural boundaries exist only as conventions that will erode over time.

## Who

- **Primary:** Backend developers — they hit the exception filter on every error path and need clear, testable per-domain error handling. They also need lint feedback when bypassing the repository pattern.
- **Secondary:** Frontend developers — import boundary enforcement prevents accidentally pulling API-only types into the web bundle.

## Constraints

- **Dependency on slices 4+5:** The `@Inject(DRIZZLE)` lint rule needs all repository adapter file paths established (slices 4+5 must be merged to staging first).
- **Exception filters are independent:** Can be implemented in parallel with lint rules.
- **No ESLint:** Project uses Biome. Custom lint rules will be Bun TS scripts integrated into CI and pre-push hooks, not ESLint.
- **RLS bypass comments are additive:** `// RLS-BYPASS: <reason>` annotations are documentation-only — no runtime behavior change.

## Out of Scope

- Migrating to ESLint or adding ESLint alongside Biome.
- Changing the repository pattern itself (established in slices 4–5).
- Modifying RLS enforcement logic — only adding annotation comments.
- New exception types or error response format changes.
